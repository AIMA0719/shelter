# PRD — Shelter (가칭 「그늘로」)

> 출발지–목적지 경로 위에서 **어디가 그늘이고 어디가 햇빛인지**를 알려주는,
> 도보·자전거를 위한 *햇빛/그늘 인지 경로 안내* 앱.

문서 버전: v0.2
작성일: 2026-06-05
상태: 기획 / 아직 코드 없음

### 확정된 핵심 결정 (v0.2)
- **그늘 엔진:** **자체 레이캐스터.** (ShadeMap API는 유료 — ㎢당 $3.99, 수목 월 단위. 서울 전역 커버 시 면적 비용 누적 + 외부 의존/라이선스 리스크 → 내재화로 확정.)
- **플랫폼:** **안드로이드 우선** (이후 iOS 확장).
- **첫 커버리지:** **서울 전역.**
- **첫 이동 모드:** **도보만.**

---

## 0. 한 줄 요약 (Elevator Pitch)

> "네이버·카카오·구글 지도는 *어디에 뭐가 있는지*는 알려주지만,
> *지금 이 길을 걸으면 얼마나 땡볕인지*는 안 알려준다.
> Shelter는 출발–도착 경로를 **그늘 비율로 색칠**하고,
> 같은 목적지라도 **가장 그늘진 길**을 추천한다."

핵심 차별점은 단 하나: **태양 위치 + 건물/가로수 높이로 그림자를 계산해, 경로의 각 구간이 출발 시각 기준 그늘인지 햇빛인지 보여준다.** 이게 되면 나머지는 다 부가기능.

---

## 1. 문제 정의 (Problem)

- 여름철 폭염·자외선·온열질환이 매년 심해짐. 보행자·자전거 이용자·유아 동반자·고령자에게 "그늘"은 안전·쾌적의 핵심 변수.
- 기존 지도 앱의 경로 안내는 **최단/최소시간**만 최적화. "덜 더운 길", "그늘진 길"이라는 축이 아예 없음.
- 사람들은 경험적으로 "이 길은 그늘져서 시원해"를 알지만, **처음 가는 길·낯선 동네에서는 알 수 없음**.
- 실제로 행정(서울시 등)도 폭염 대응으로 그늘막·쿨링로드를 만들지만, 시민이 *실시간으로 그늘 경로를 탐색*할 수단은 없음.

### 누가 가장 절실한가 (Desperate users)
1. **여름 출퇴근·통학 보행자** — 매일 같은 더위를 겪고, 1℃·1분의 그늘에 민감.
2. **유아차/어린이 동반 보호자** — 직사광선 회피가 거의 의무.
3. **자전거·도보 운동/산책러** — 루트 선택 자유도가 높아 그늘 최적화 여지 큼.
4. **택배·배달·현장 노동자** — 폭염 노출이 직업 안전 문제.
5. **관광객** — 낯선 도시에서 더위 피하며 걷고 싶음.

## 2. 목표 & 비전

### 비전
"날씨를 *피하는* 길찾기." 1차는 그늘이지만, 장기적으로는 **체감 쾌적도(thermal comfort) 라우팅** 플랫폼으로 확장 — 그늘 + 바람 + 수변 + 미세먼지 + 경사까지 통합한 "가장 덜 힘든 길".

### 1년 목표 (가정)
- 서울 전역에서 도보 경로에 대해 그늘 비율을 90%+ 정확도로 시각화.
- "가장 그늘진 경로" 추천이 기본 경로 대비 그늘 비율을 유의미하게(예: +15%p) 높임.
- 여름 시즌 재방문율(주 3회+) 유의미 코호트 확보.

### 비목표 (Non-goals, MVP에서 안 함)
- 실내 길찾기, 대중교통 환승 안내, 내비게이션 음성 턴바이턴(후순위).
- 전국 동시 커버리지(서울→수도권→광역시 순으로 확장).
- 자체 지도 타일 렌더링 엔진 신규 개발(기존 지도 SDK 활용).

## 3. 핵심 인사이트 — "그늘을 어떻게 아는가?" (제품의 심장)

이게 PRD에서 제일 중요한 부분. 그늘 = **태양의 위치**와 **빛을 가리는 물체(건물·나무·지형)**의 기하학적 관계.

### 3.1 태양 위치 (공짜, 쉬움)
- 위도·경도·날짜·시각만 있으면 **태양 고도각(altitude)·방위각(azimuth)**을 정밀 계산 가능.
- 알고리즘: NOAA Solar Position / SPA, 라이브러리 `suncalc`(JS), `pvlib`(Py), `SolarPositionAlgorithm`.
- 비용 0, 데이터 불필요, 오차 무시 가능. **여기는 문제 없음.**

### 3.2 그림자를 드리우는 물체 (어려움 — 데이터가 관건)
| 캐스터 | 필요 데이터 | 한국 데이터 소스 |
|---|---|---|
| **건물** | 건물 외곽(footprint) + 높이/층수 | 국가공간정보포털(NSDI), V-World(브이월드) 건물통합정보, 도로명주소 건물DB, OSM `building`/`building:levels` |
| **지형** | 수치표고모델(DEM) | 국토지리정보원 DEM(5m/30m) |
| **가로수/수목** | 수목 위치·수고·수관폭 | 서울시 가로수 현황 오픈데이터, 지자체 가로수 DB, 위성 NDVI, LiDAR 수관고(고급) |
| **고가/구조물** | 육교·고가도로·정류장 캐노피 | OSM, 시설물 DB(부분적) |

> **MVP 현실 판단:** 건물만으로도 도심 그늘의 대부분을 설명할 수 있다. **MVP = 건물 그림자만.** 가로수는 Phase 2(정확도 체감이 크지만 데이터 정제 비용 큼).

### 3.3 그림자 계산 방식 (3가지 후보)
1. **레이캐스팅(on-the-fly):** 경로 위 각 샘플 점에서 태양 방향으로 광선을 쏴, 건물 높이가 그 점을 가리는지 판정. 경로 단위라 가벼움. → **MVP 추천.**
2. **그림자 래스터 사전계산(shadow map):** 도시를 시각별로 렌더링해 "그림자 유무" 래스터 타일 생성 후 경로에서 샘플링. ShadeMap.app 방식. 정밀하지만 인프라 무거움. → Phase 2/3.
3. **하이브리드:** 건물 높이맵(heightmap raster)을 타일로 보관 + 요청 시 경로 bbox만 GPU/CPU로 그림자 캐스팅. → 확장 시 정답.

### 3.4 선행 사례 (Prior art — 바퀴 재발명 금지)
- **ShadeMap (shademap.app):** 전 세계 건물+지형 기반 시각별 그림자 맵. **API 제공.** → MVP를 *이 위에서 부트스트랩*하고, 한국 특화(가로수·자전거·한국 지도 UX)로 차별화하는 전략이 현실적.
- 학술: LA·취리히 등 "shade routing / cool path" 연구 다수(그늘·수목 가중 최단경로).
- 시사점: 핵심 기술은 *가능함이 증명됨*. 승부는 **한국 데이터 정합 + 경로 추천 UX + 자전거/도보 특화**.

## 4. 핵심 사용자 시나리오 (User Stories)

1. **(시각화)** "지금 회사→집까지 걷는데, 어디가 그늘인지 색으로 보고 싶다."
   → 경로 폴리라인이 **초록(그늘)/노랑(햇빛)** 그라데이션, 상단에 "이 경로 그늘 62%".
2. **(추천)** "조금 돌더라도 제일 그늘진 길로 가고 싶다."
   → '최단' vs '가장 그늘짐' vs '균형' 3가지 경로 카드 비교(거리/시간/그늘%/예상 체감).
3. **(시간 선택)** "오후 3시에 출발할 건데 그때 기준으로 보여줘."
   → 출발 시각 슬라이더. 태양 위치 재계산. 긴 경로는 *이동 중 태양이 움직이는 것*까지 반영.
4. **(자전거)** "자전거로 한강까지, 그늘+안전 도로 우선."
   → 자전거 모드: 그늘 가중 + 자전거도로 가중.
5. **(저장/알림)** "매일 출근길 그늘 좋은 시간대 알려줘."
   → 즐겨찾기 경로 + "오늘 8:10 출발이 8:40보다 그늘 18%p 좋아요" 알림(Phase 2+).

## 5. 기능 명세 (Features)

### MVP (P0 — 이것만 되면 출시)
- [ ] 출발지/목적지 입력(검색·지도 롱프레스·현위치).
- [ ] **단일 경로의 그늘/햇빛 시각화** — 구간별 그늘 판정 → 색칠 폴리라인.
- [ ] 경로 요약: 총거리·예상시간·**그늘 비율 %**.
- [ ] 출발 시각 선택(지금 / 시간 슬라이더).
- [ ] 도보 모드(자전거는 P1).
- [ ] **서울 전역** 커버(건물 그림자 기준). 단, Phase 0 정확도 검증은 한 권역(예: 강남)에서 선행.
- [ ] **안드로이드** 출시(iOS는 이후).

### P1 (출시 직후)
- [ ] **'가장 그늘진 경로' 추천** + 최단/균형 비교(라우팅 가중치 적용).
- [ ] 자전거 모드.
- [ ] 가로수 그림자 반영(서울시 가로수 데이터).
- [ ] 즐겨찾기 경로, 최근 검색.
- [ ] 자외선지수·기온·폭염특보 배지(기상청 API).

### P2+ (확장)
- [ ] 출발 시간 추천("언제 나가면 가장 시원").
- [ ] 체감 쾌적도 라우팅(바람·수변·미세먼지·경사 통합).
- [ ] 그늘막/쿨링쉼터/음수대 POI 오버레이(지자체 데이터).
- [ ] 겨울 모드 반전("햇빛 많은 길" — 추울 때).
- [ ] 커뮤니티 보정(사용자가 "여기 실제론 그늘" 피드백 → 데이터 개선).
- [ ] B2B/공공: 지자체 폭염 취약경로 분석 대시보드.

## 6. 기술 아키텍처 (제안)

```
[모바일 앱]
  - 지도 SDK: 네이버/카카오 Maps SDK (한국 도보 데이터 강점) 또는 MapLibre+커스텀 타일
  - 경로 폴리라인 색칠 렌더링, 시각 슬라이더, 경로 비교 카드
        │  (출발/도착, 모드, 출발시각)
        ▼
[백엔드 API]
  1) 경로 후보 획득
     - MVP: 카카오/네이버/구글 Directions(도보·자전거)에서 경로 좌표 받기
     - P1+: 자체 라우팅 엔진(Valhalla/GraphHopper/OSRM) + OSM 그래프
            → 엣지 가중치 = f(거리, 그늘%) 로 '그늘 최적 경로' 직접 생성
  2) 그늘 판정 (핵심 엔진)
     - 경로를 N m 간격으로 샘플 → 각 점 (위경도, 도착예상시각)
     - 태양위치(suncalc/SPA) 계산
     - 건물 높이 데이터로 레이캐스팅 → 그늘/햇빛 + (선택)신뢰도
     - MVP 부트스트랩 옵션: ShadeMap API 타일 샘플링
  3) 응답: 구간별 [좌표, shade=true/false, 신뢰도], 경로 요약 통계
        │
        ▼
[데이터 레이어]
  - 건물 height raster / footprint (V-World, NSDI, OSM) — 전처리해 타일/공간DB(PostGIS) 적재
  - DEM(지형), 가로수(P1), POI(쉼터/음수대, P2)
  - 캐시: (타일 x 날짜버킷 x 시각) 그림자 결과 캐싱으로 비용 절감
```

### 스택 제안 (확정 결정 반영)
- **앱:** **안드로이드 우선.** 네이티브 Kotlin(Jetpack Compose) 또는 Flutter(안드 먼저 빌드/배포 후 iOS 재사용). iOS 확장 의향이 크면 Flutter, 안드 품질 우선이면 Kotlin.
- **백엔드:** Python(GIS 친화 — `shapely`, `rasterio`, `pvlib`/`suncalc`, `pyproj`) 또는 Node + 공간연산 분리.
- **공간 DB:** PostGIS.
- **라우팅:** Valhalla(보행 프로파일 + 커스텀 코스팅; 자전거는 P1).
- **그림자:** **자체 레이캐스터(numpy/raster) → 추후 GPU.** (서울 전역 건물 높이맵을 타일로 전처리·적재.)

## 7. MVP 범위를 더 줄인다면 (Smallest shippable)

> **"서울 강남 일대, 도보, 지금 시각 기준, 단일 경로 그늘 색칠 + 그늘% 한 줄."**
> 라우팅 추천도 빼고, 데이터도 한 권역 건물만, 시각화 하나로 *가치 가설*을 검증.
> 가설: "경로의 그늘을 눈으로 보여주기만 해도 사람들이 길을 바꾸고 다시 온다."

## 8. 데이터·법무 체크리스트 (착수 전 반드시 확인)

## 9. 리스크 & 가정

| 리스크 | 영향 | 완화 |
|---|---|---|
| 건물 높이 데이터 품질·누락 | 그늘 오판 → 신뢰 추락 | 신뢰도 표기, 권역 한정 출시, 커뮤니티 보정 |
| 지도 API 약관 위반 가능성 | 서비스 중단 | 착수 전 약관 검토, 필요시 MapLibre+자체타일로 전환 |
| 그림자 계산 비용/지연 | UX 저하 | 타일·시각 버킷 캐싱, 경로 bbox만 계산 |
| 가로수·골목 미세 그늘 미반영 | 체감 정확도 한계 | P1에서 가로수 추가, "근사치" 명시 |
| 계절성(여름만 쓰는 앱) | 리텐션 | 겨울 '햇빛 모드', 상시 쾌적도 라우팅으로 확장 |
| 선행자(ShadeMap 등) | 차별화 약화 | 한국 데이터·자전거/도보 라우팅·한국 UX로 수직 특화 |

### 핵심 가정 (검증 필요)
1. 사람들은 *조금 돌아가더라도* 그늘진 길을 선택할 의향이 있다.
2. 건물 그림자만으로도 체감상 "맞네"라고 느낄 정확도가 나온다.
3. 여름 외 시즌에도 쓸 이유를 만들 수 있다.

## 10. 성공 지표 (Metrics)

- **북극성:** 주간 활성 사용자 중 "그늘 경로를 실제로 따라간 비율"(추천 채택률).
- 그늘 시각화 → 경로 변경 전환율.
- 추천 경로의 평균 그늘% 개선폭(vs 최단).
- 여름 시즌 주 3회+ 재방문 코호트.
- 그늘 판정 정확도(현장 샘플/사용자 피드백 대비).

## 11. 로드맵 (단계)

- **Phase 0 — 타당성(2~4주):** 태양위치+건물 레이캐스터 프로토타입(노트북/스크립트)으로 강남 한 블록 그늘을 실제 사진과 대조 검증. 지도/데이터 약관 검토.
- **Phase 1 — MVP(6~10주):** 단일 경로 그늘 시각화 앱, 서울 일부 권역, 도보.
- **Phase 2 — 추천(+6~8주):** 그늘 최적 라우팅, 자전거, 가로수, 기상 배지.
- **Phase 3 — 확장:** 시간 추천, 쾌적도 라우팅, 쉼터 POI, 권역 확대, B2B.

## 부록 A — 왜 기존 지도는 이걸 안 하나
그들은 *범용 최단/최소시간*에 최적화돼 있고, 그림자는 (a) 정밀 건물 높이 데이터, (b) 시각·계절 종속 계산, (c) 좁은 사용자 가치(여름·보행)라 우선순위가 낮다. 바로 그 틈이 버티컬 기회다.

## 부록 B — 확장 한 줄
"Shade routing"은 시작일 뿐. 진짜 시장은 **"날씨를 피하는 길찾기"** — 더위·비·바람·미세먼지·언덕을 피해 *가장 덜 힘든 길*을 주는, 보행자/마이크로모빌리티용 쾌적도 내비게이션.
